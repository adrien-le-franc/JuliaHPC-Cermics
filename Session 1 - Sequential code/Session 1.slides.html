A.1) What is Julia?¶

Maybe the solution to the two-language problem (https://www-nature-com.extranet.enpc.fr/articles/d41586-019-02310-3):

  • User-friendly syntax for high-level programming
  • C-level speed (when done right) for high-performance computing

B. Measuring code performance¶

B.0) Running example: shortest paths in graphs¶

Let $G = (V, E, d)$ be an undirected weighted graph, and $s, t$ be two vertices. Computing a shortest $s-t$ path in $G$ can be done with Dynamic Programming.

If $c_k(v)$ denotes the cost of a shortest $s-v$ path using at most $k$ edges, then:

\begin{align} c_0(v) & = \begin{cases} 0 & \text{if $v = s$} \\ +\infty & \text{otherwise} \end{cases} \\ c_{k}(v) & = \min \{c_{k-1}(v)\} \cup \{c_{k-1}(u) + d_{uv} ~|~ u \in \mathcal{N}(v)\} \end{align}

And a shortest path has length at most $n-1$, so we want to compute $c_{n-1}(t)$

In [4]:
function bellmanford1(G; s = 1, t = nv(G), d = ones(Float64, nv(G), nv(G)))
    n = nv(G)
    c = []
    c0 = [(v == s ? 0 : Inf) for v = 1:n]
    push!(c, c0)
    for k = 2:n
        ck = [c[k-1][v] for v = 1:n]
        for v = 1:n
            if length(neighbors(G, v)) > 0
                ck_by_u = minimum([c[k-1][u] + d[u, v] for u in neighbors(G, v)])
                ck[v] = min(ck[v], ck_by_u)
            end
        end
        push!(c, ck)
    end
    return c[n][t]
end

function bellmanford2(G; s = 1, t = nv(G), d = ones(Float64, nv(G), nv(G)))
    n = nv(G)
    c = Matrix{Float64}(undef, n, n)
    for v = 1:n
        c[1, v] = (v == s ? 0. : Inf)
    end
    for k = 2:n
        for v = 1:n
            c[k, v] = c[k-1, v]
            for u in neighbors(G, v)
                c[k, v] = min(c[k, v], c[k-1, u] + d[u, v])
            end
        end
    end
    return c[n, t]
end

C.1) General advice¶

  • Premature optimization is the root of all evil
  • Never optimize before profiling, you could end up improving a useless part of the code
  • Use https://github.com/JunoLab/Traceur.jl (slow!) to find inefficiencies in your code

C.2) Memory allocations¶

  • Prefer in-place operations
  • Pre-allocate output memory
  • Loops are not slower in Julia... most of the time

The first version of Bellman-Ford creates a new array each time it computes a minimum: SAD!

This is why most Julia functions (especially in packages like LinearAlgebra) have two versions:

func!(x) = ...
func() = begin; x = ...; func!(x); return x; end

That's the way it should be.

C.3) Typing¶

The compiler is your friend, treat it as such¶

  • Julia is fast when it compiles type-specific code, ie when type inference works
  • Avoid abstract types in strategic places: container initializations, field declarations
  • Don't change variable types when you don't have to
  • Use @code_warntype to debug
